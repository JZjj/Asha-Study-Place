Distributed Machine Learning for Training and Inference

When one device isn't enough: splitting data, models, and inference across many nodes — federated learning on the edge, data- and model-parallel training in the datacenter, and collaborative inference across cloud, edge, and end devices. The recurring cost is communication.

01

Federated learning & FedAvg

Train a shared model across many devices that keep their data local. The server only ever sees model updates, never the raw samples — privacy by construction.

The setting

Each edge device i holds its own examples (xi, yi) and trains locally. Only gradients (or weight deltas) are shipped to a central node, which averages them. The global objective is the usual empirical risk, but the data lives on the devices:

Σi=14 ‖yi − F(xi)‖²global loss over device data
∇w = (∇w1 + ∇w2 + ∇w3 + ∇w4) / 4averaged gradient
Why federated learning

The data never leaves the device. A powerful DNN can be trained across millions of phones without any user sharing their photos, keystrokes, or health data. Privacy is the whole point — the trade is that the data is now non-IID, devices are heterogeneous, and the link is lossy (see §2).

Source: McMahan et al., "Communication-Efficient Learning of Deep Networks from Decentralized Data," AISTATS (PMLR) 2017.

FedAvg — the four-step round McMahan 2017

One communication round of Federated Averaging repeats four steps:

StepWhat happensUpdate
1 · BroadcastServer initializes a global model and sends it to every participating node.wi = wglobal for each i
2 · Local trainEach node trains on its own data for a few epochs. Local training length may vary.w′i = minwi L(F(Di), Yi)
3 · Report deltaEach node sends its local weight change back up to the central node.Δwi = w − wi
4 · AggregateServer averages the deltas to update the global model, then loops.w + (1/N) Σi Δwi
Key idea

FedAvg pushes several local SGD steps before each communication round, not one gradient per round. That amortizes the (expensive) uplink over many cheap local updates — the "communication-efficient" in the paper's title.

02

Federated learning: the four problems (and fixes)

FL differs from datacenter distributed training in one crucial way: you don't control the clients. Their data, speed, link, and trust all vary. Four failure modes follow, each with a proposed remedy.

Problem 1 — Non-IID data

Unlike datacenter training where data is shuffled randomly (IID), each device's local data is highly biased — one phone sees only dogs, another only cats. Averaging updates from conflicting local optima can badly degrade the global model.

How bad

In extreme 1-class non-IID (each client holds a single label), test accuracy collapses far below the IID curve on both MNIST and CIFAR-10 across 500 communication rounds. 2-class non-IID is milder but still hurts. (Zhao et al., 2018.)

Fix · data sharing

Zhao et al. propose seeding every device with a small globally-shared subset of data. Just 5% shared data raises CIFAR-10 accuracy by ~30% — a little common ground re-aligns the local optima.

Problem 1b — Non-IID via a proximal term FedProx

FedProx keeps each device's local solution from drifting too far from the current global weights by adding an ℓ2 proximal penalty to the local objective:

minw hk(w; wt) = Fk(w) + (μ/2)‖w − wt‖²local objective + proximal term
  • The extra term minimizes the distance between the initial weight wt and the learned weight w.
  • Guarantees the local update stays close to the global model → stable aggregation under heterogeneity. Devices may return a γk-inexact minimizer (partial local work is fine). (Li et al., MLSys 2020.)

Problem 2 — System heterogeneity

Edge devices differ in processing speed. Because the round can't finish until the slowest device reports, total latency is bottlenecked by the straggler → slow convergence.

Fix · HeteroFL

Instead of forcing every device to train the whole model, HeteroFL assigns each device a sub-network sized to its compute. Weak devices train a small slice of the global parameters Wg; strong devices train more. Nested parameter blocks let the server still aggregate them into one model. (Diao et al., 2020.)

Problem 3 — Communication

The device↔cloud link can drop or corrupt updates, hurting both latency and accuracy. The remedy is to transmit fewer, more useful updates.

CMFL scores each local update's relevance to the global direction by how often its per-coordinate sign agrees with the global update's sign:

e(u, ū) = (1/N) Σj=1N 𝕀(sgn(uj) = sgn(ūj))fraction of matching signs
  • uj is the sign of the weight after the local update; ūj is the global reference.
  • Only devices with high relevance transmit; irrelevant updates are excluded, cutting communication overhead. (Luping, Wei & Bo, ICDCS 2019.)

Problem 4 — Privacy

Gradient leakage

Even without raw data, an attacker who sees the transmitted gradients can reconstruct the original training inputs. "Data never leaves the device" is necessary but not sufficient — the gradients themselves leak.

Client selection as an RL problem FedMARL

Which clients should participate each round? FedMARL frames it as a multi-objective optimization over a client-selection matrix A = [ant], trading final accuracy against training latency and bandwidth:

maxA 𝔼[ w1·Acc(T) − w2·Σt∈T Ht − w3·Σt∈T Bt ]accuracy − latency − bandwidth
  • Acc(T) = final model accuracy · Σ Ht = total training latency · Σ Bt = total bandwidth.
  • Weights w1, w2, w3 are set by the FL application designer. The problem is hard to solve directly, so it is cast as multi-agent reinforcement learning (MARL).
  • Result: FedMARL beats both random client dropping and plain FedAvg on VGG6 / CIFAR-10 (random dropping already beats FedAvg). (Zhang, Lin & Zhang, AAAI 2022.)
03

Distributed training: data parallelism

Replicate the whole model on every node, split the batch across them. Each node computes gradients on its shard; then all gradients are synchronized. Same as FL, but the data is now IID.

Batchify, then synchronize

With learning rate η and a mini-batch B of size n per node:

L(w) = (1/|X|) Σx∈X l(x, w)loss function
wt+1 = wt − η·(1/n) Σx∈B ∇l(x, wt)weight update

Example: 4 GPUs each holding a local batch of 64 give a total batch size of 256. The only question is how the per-node gradients get summed — this is where the communication schemes differ.

Parameter server Li et al., OSDI 2014

A central server holds the authoritative weights. Workers compute gradients, push them to the server, the server aggregates and pushes updated weights back.

Total communication = 2(N−1)·GN = #nodes, G = gradient size
Not scalable

Fault-tolerant (if a worker dies, others continue) — but the central node is a bottleneck. As N grows it cannot handle traffic from all workers at once. The whole 2(N−1)G volume funnels through one node.

All-reduce

Remove the central server: have the nodes sum gradients among themselves. Each round is four stages — forward pass (local mini-batch) → backward pass (local gradients) → all-reduce (sum gradients across all nodes, broadcast the sum) → parameter update (each node applies the aggregated gradient to its own replica).

Why all-reduce beats a parameter server

There is no single aggregation node, so no single bottleneck. Every node does an equal share of the communication and ends up with the identical summed gradient. The trick is scheduling the messages so the link load is balanced — that's ring all-reduce.

Ring all-reduce

Arrange the N nodes in a ring. Split each node's gradient into N chunks. The algorithm runs in two phases, each of N−1 steps, where every node only ever talks to its neighbor:

  • Share-reduce (scatter-reduce) phase: at each step a node passes one chunk to its neighbor, which adds it to its own. After N−1 steps, each node holds one fully-reduced chunk (the sum of that chunk across all nodes).
  • Reduce-only (all-gather) phase: circulate those completed chunks around the ring so every node ends up with all fully-reduced chunks.
Total communication = 2(N−1)·Gsame volume as PS…
Per-node bandwidth = 2(N−1)/N · G…but split evenly across nodes
Bandwidth-optimal

Ring all-reduce moves the same 2(N−1)G total, but spreads it uniformly — each node sends/receives only 2(N−1)/N·G, which approaches 2G and is independent of N. No node is a bottleneck, so unlike the parameter server it scales. Each node carries an identical workload.

Communication reduction: Deep Gradient Compression Lin 2017

As node count grows, communication starts to dominate computation on the timeline. DGC shrinks the payload by sending only the important gradients:

  • Accumulate each layer's gradient locally; transmit a coordinate only when its magnitude exceeds a threshold (top s%).
  • Sub-threshold gradients are kept and accumulated (via a mask) for later rounds, so no information is lost — just delayed.
  • The resulting sparse gradient is compressed with run-length encoding before the all-reduce: Gt ← Σk encode(G̃tk).

Effect on the timeline: with more nodes, communication grows to overlap or exceed computation; DGC shrinks the communication bar back down so compute dominates again.

04

Distributed training: model parallelism

When the model itself is too big for one device, split the model across devices — put different layers on different accelerators (pipeline parallelism). The hazard is idle time from sequential dependencies.

Naive model parallelism wastes hardware

Place layers on devices 0–3. The forward pass must reach device 3 before the backward pass can start, and each device is idle while the others work. This severe under-utilization comes directly from the network's sequential dependency.

GPipe — micro-batches Huang 2019

GPipe splits every mini-batch of size N into M equal micro-batches, so different accelerators can work on different micro-batches simultaneously — like an assembly line.

The bubble

At the start and end of each batch the pipeline isn't full: some devices sit idle. This idle region is the pipeline bubble. More micro-batches (larger M) shrink the bubble's relative cost, since the steady-state region grows while the fixed fill/drain stays the same.

PipeDream — keep the pipe full Narayanan 2019

PipeDream enables faster DNN training by combining intra-batch parallelism with inter-batch parallelism. Instead of finishing one batch before starting the next, workers interleave forward and backward passes of different batches (a "1F1B"-style schedule).

  • A startup state fills the pipeline; then a steady state runs where every worker alternates forward and backward passes with almost no idle time.
  • All inputs in flight use weights from the last flush; a periodic pipeline flush adds up gradients — the price paid to keep the pipe busy is weight staleness that must be managed.
GPipe vs PipeDream

Both attack the bubble. GPipe keeps synchronous semantics (flush every mini-batch) and reduces the bubble with many micro-batches. PipeDream interleaves batches to eliminate idle time in steady state, trading exact synchrony for throughput.

05

Distributed & collaborative inference

Not every input needs the full network, and not every layer needs the same device. Three ideas: exit early on easy inputs, split the network across cloud/edge/device, and partition the feature maps across an edge cluster.

BranchyNet — early exit Teerapittayanon 2016

Data samples aren't equally hard. Easy CIFAR-10 images can be classified correctly after just a few layers; hard ones need the whole network. BranchyNet attaches side exits along the backbone and stops as soon as one is confident.

entropy(y) = Σc∈C yc log ycconfidence score at an exit
  • At each exit compute the entropy of the softmax output. Low entropy = confident → emit the result there and skip the rest of the network for faster inference.
  • Train all exits jointly by summing the cross-entropy loss at every exit point, each weighted by wn:
Lbranchynet(ŷ, y; θ) = Σn=1N wn L(ŷexitn, y; θ)jointly-trained multi-exit loss

DDNN — cloud, edge, and end devices Teerapittayanon 2017

Distributed DNNs place the early-exit idea over a computing hierarchy: end devices → edge (fog) → cloud. Three configurations: (a) cloud-only, (b) cloud + device, (c) cloud + geographically distributed devices.

  • Each end device runs a small local DNN; a local aggregator combines their outputs for a fast local exit.
  • If the local exit isn't confident (η(x) < T), the activations after the last conv layer are sent up to the cloud aggregator for the full model — computation offloaded only when needed.
Why send activations, not raw data

The local conv layers act as a learned, compact feature extractor. Forwarding their activation output (not the raw image) to the cloud saves bandwidth and avoids shipping the raw input — echoing the FL privacy theme.

ADCNN — partitioning conv across an edge cluster Zhang 2020

For a single big conv layer, split the work across nearby edge devices. Two ways to partition, with very different communication:

PartitionHow it splitsCommunication cost
ChannelwiseEach node owns a subset of channels (C/2) and all of H×W.High — nodes must exchange partial output feature maps to sum them.
SpatialSplit the input in the spatial dims (H/2 × W/2 tiles); weights are duplicated on every device.Only the data halo — the thin border of pixels each tile needs from its neighbors.
Data halo → FDSP

In spatial partition, a tile needs a border strip from adjacent tiles to compute conv outputs at its edges — the data halo — which still costs cross-tile transmission. FDSP (Fully Decomposable Spatial Partition) eliminates it entirely by padding each tile's edge pixels with zeros, so tiles compute completely independently. No cross-tile transfer at all.

  • Motivation: profiling VGG16 shows earlier conv layers take far longer than later ones — so those are the layers worth distributing.
  • Pipeline: Step 1 — progressively retrain the CNN into a distributable form; Step 2 — at inference, split the input into tiles, process each on a conv node, and combine at a central node.
  • On 9 Raspberry Pis (8 conv nodes + 1 central), ADCNN cuts average latency by 6.68× vs a single device and 4.42× vs remote cloud, across FCN, YOLO, VGG16, ResNet34, CharCNN.

Exam recap

FedAvg loop

Broadcast wi=wglobal → local train → send Δwi=w−wi → aggregate w+(1/N)ΣΔwi. Data stays on-device.

FL's four problems

Non-IID (fix: data-sharing, FedProx), heterogeneity (HeteroFL), communication (CMFL), privacy (gradient leakage).

FedProx

Add proximal term (μ/2)‖w−wt‖² to keep local updates near the global model under non-IID.

Parameter server

Comm = 2(N−1)G, all funneled through one node → central bottleneck, not scalable.

Ring all-reduce

Share-reduce + reduce-only phases. Same 2(N−1)G total but per-node 2(N−1)/N·G → bandwidth-optimal, scales with N.

Model parallelism

GPipe: split into M micro-batches to shrink the bubble. PipeDream: interleave intra + inter-batch to fill the pipe.

Gradient compression

DGC sends only gradients above a magnitude threshold, accumulates the rest, RLE-compresses — cuts communication.

Distributed inference

BranchyNet exits early on low-entropy inputs; DDNN offloads to cloud only if unconfident; ADCNN tiles conv spatially (FDSP zero-pads to kill the halo).
