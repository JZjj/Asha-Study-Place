AI Compilers & the CNN Accelerator

Two halves of the same story: the software path that lowers a Python model into optimized device code, and the hardware path — MAC arrays, memory hierarchies, and systolic arrays — that runs it while paying as little for memory as possible.

01

The ML software–compiler stack

A model written in Python does not run on silicon directly. It is lowered through five layers, each converting a higher-level description into something closer to the metal.

LayerExamplesJob
ML FrameworksPyTorch, TensorFlow, Chainer, Caffe, TheanoHigh-level interfaces where models are defined, trained, and debugged.
IR & Graph OptimizationTorchDynamo, FX, ONNXConverts model definitions into an analyzable graph IR for optimization.
ML Compiler / Code GenTVM, Triton, ONNXTranslates the IR into optimized kernels and executable code for a specific device.
Kernel LibrarycuDNN, cuBLAS, CUTLASSExecutes optimized kernels, manages device memory, provides primitives.
Hardware BackendCPU, GPU, AI AcceleratorThe physical devices that execute the compiled, optimized model.
Why an IR at all

The framework layer is optimized for human ergonomics; hardware wants regular, analyzable computation. The IR is the middle ground — a graph the compiler can rewrite, fuse, and re-order before any code is generated. The rest of the lecture is really about the two ends of this stack: the graph on top and the accelerator on the bottom.

02

Frameworks & automatic differentiation

Frameworks split into two execution philosophies, and the choice ripples all the way down to how easy graph optimization becomes.

Graph mode vs eager mode

Graph mode

Expose a graph-building API: the user first constructs a computation graph, then later executes it. Theano, Caffe, early TensorFlow.

Eager mode

Operations execute immediately as they are called in Python, rather than being added to a static graph. PyTorch, Chainer, TensorFlow 2.

The trade-off

Eager mode is easy to write and debug, but the downside is that it makes graph-level optimization harder for the compiler — there is no static graph sitting around to rewrite. Graph mode is the opposite: harder to author, but the whole computation is visible up front.

PyTorch: "everything is just a program" inherits Chainer

  • The DNN model is programmed: the API lets the user freely select the layer configuration (nn.Module, forward()).
  • This philosophy is not limited to models — it applies to optimizers and data loaders as well.
  • Bidirectional exchange with external libraries: convert between NumPy arrays and tensors via torch.from_numpy() and .numpy().
  • Ships an automatic differentiation engine that performs the gradient computation for you.
x = torch.tensor(2.0, requires_grad=True); y = x*2 + x*3y.backward() → x.grad

Autograd records a graph: y = (2x) + (3x), each product a node. Calling y.backward() walks it in reverse and deposits the gradient in x.grad.

Control flow vs data flow

A strict separation

PyTorch keeps control flow (branches, loops) apart from data flow (tensors and their ops). Control flow is resolved by Python + optimized C++ on the host CPU, producing a linear sequence of operator invocations on the device. Operators then run on CPU or GPU. Memory is first allocated on the CPU, which manages the allocation and mapping onto the GPU.

What if the function isn't differentiable?

Autograd needs a gradient. When the op has none (e.g. a hard threshold), the options are:

  • Subgradients / piecewise derivatives — e.g. y = |x| at x=0 yields x.grad = 0.
  • Approximation gradients — substitute a smooth surrogate.
  • Customized gradient definition — write your own backward.
  • Compilation error — if none is provided, autograd cannot proceed.
Straight-Through Estimator (STE)

For a binary step y = (x > 0).float() the true gradient is 0 almost everywhere — useless. STE approximates the gradient as 1 (passes it straight through), so training signal still flows. In a custom torch.autograd.Function, the backward returns a surrogate such as grad_output * (|x| ≤ 1).

TensorFlow: the static-graph counterpoint

  • Open-source framework by Google (2015); supports CPU, GPU, and TPU acceleration across CNNs, RNNs, Transformers, RL, and scientific computing.
  • Builds a static computation graph (predefined dataflow): nodes = operations, edges = data (tensors).
  • Because the whole graph is known, it is optimizable and deployable across hardware, and enables parallelism and graph-level optimizations.
z = x * y → Tensor("mul:0", shape=(), dtype=float32)nothing runs — graph only

In TF1 z is just a graph node until sess.run(z) materializes it (12.0). In TF2's eager mode the same line executes immediately and prints tf.Tensor(12.0, …).

03

Graph-level optimization & kernels

Once the computation is a graph, the compiler can rewrite it into a cheaper but numerically equivalent graph — then hand the pieces to hand-tuned kernel libraries.

The canonical rewrite

y = x*2 + x*3 ≡ y = x*(2+3)2 mults + 1 add → 1 add + 1 mult

The left graph has two multiply nodes feeding an add. The right graph folds 2+3 into a constant 5 and does a single multiply — same result, fewer operations. This is exactly the kind of rewrite eager mode makes harder, because the graph must be recovered before it can be transformed.

torch.fx capture & transform

torch.fx.symbolic_trace(model) captures the forward() as an explicit graph of nodes — placeholder, call_module, call_function — that you can inspect and rewrite:

%relu_1 = call_function[target=relu](args=(%fc1,)) → %relu_2 = call_function[target=relu](args=(%relu_1,))two ReLUs in a row = redundant
Concrete pass: remove redundant ReLU

Walk gm.graph.nodes; when a call_function node is relu and the previous op was also relu, skip it (continue), copying every other node into a new graph. Wrapping the result in torch.fx.GraphModule yields an optimized model. torch.fx gives the flexibility to modify and transform the computation graph, enabling performance and efficiency improvements.

Triton Python that compiles to GPU

  • Open-source compiler and language (originally Harvard/OpenAI, now integrated into PyTorch). Writing efficient GPU kernels in raw CUDA is complex and error-prone.
  • Lets you write custom GPU kernels in Python and reach CUDA-level performance with far simpler code — researchers often need custom fused kernels beyond what cuDNN/cuBLAS offer.
  • Automatic tiling & vectorization for performance portability, and fusion-friendly: integrates with PyTorch's graph optimizers.
  • A @triton.jit matmul kernel uses tl.program_id, tl.arange, tl.load, tl.dot, tl.store — Python-like syntax with compiler-grade optimization.

The division of labor

cuDNN accelerates deep-learning-specific ops; cuBLAS accelerates general linear algebra. Both are critical layers of the GPU software stack that make frameworks like TensorFlow and PyTorch fast.

04

AI accelerators: the hardware landscape

GPUs win on throughput but are general-purpose and power-hungry. Specializing the silicon trades flexibility for energy efficiency — that trade is the whole design space.

Why not just use CPUs and GPUs?

  • GPU > CPU for both training and inference throughput — it leverages a highly parallel architecture and reaches 10×–20× higher throughput than a CPU.
  • But GPUs are general purpose, have high power consumption and latency, and do not support sophisticated pruning and quantization algorithms in hardware.
  • ASIC implementations accelerate inference: Google TPU, Apple Neural Engine, Cerebras AI chip.
  • FPGA accelerators offer good programmability & flexibility, short development cycles, and serve as a benchmark before committing to an ASIC (e.g. Xilinx Alveo card).

The flexibility ↔ efficiency spectrum

Read the spectrum

ASIC offers the highest energy efficiency (TOPS/W) but is only suitable for specific applications. CPU is fully general-purpose but has the lowest energy efficiency. CGRA (coarse-grained reconfigurable array) and FPGA sit in between — reconfigurable fabrics that recover some efficiency without hard-wiring one function.

Anatomy of an AI accelerator

The accelerator is one device on a Network-on-Chip (NoC) alongside the CPU and GPU, all sharing DRAM. It executes the part of the machine code related to the AI workload — a Python conv can be compiled to a bitstream and dispatched to the accelerator.

  • DRAM — large off-chip memory (~GB).
  • Register file (RF) and on-chip Weight SRAM / Activation SRAM — staging buffers close to compute.
  • Compute core — a Matrix MAC array for 2D matrix multiply, a Vector MAC, and a Special Function Unit (activations, etc.).

The energy hierarchy why memory dominates

Every operand read costs energy, and cost explodes as you move away from the ALU. Normalized to a single ALU operation:

AccessSizeNormalized energy
ALU (compute)—1× (reference)
Register file (RF)0.5–1.0 kB1×

The 200× rule

A DRAM access costs roughly 200× a compute op. This single fact drives nearly every accelerator design decision: the arithmetic is cheap; moving the data is what you pay for. Maximizing on-chip data reuse to avoid DRAM trips is the central optimization.

05

Matrix multiply on the accelerator: dataflow & reuse

Walking a 2×2 matmul through the hardware shows exactly where memory traffic comes from — and how reordering the computation cuts it without changing the result.

[W₀₀ W₀₁; W₁₀ W₁₁] × [X₀₀ X₀₁; X₁₀ X₁₁] = [Y₀₀ Y₀₁; Y₁₀ Y₁₁]Yᵢⱼ = Σ Wᵢ·Xⱼ

Each output Y₀₀ = W₀₀X₀₀ + W₀₁X₁₀ is built by streaming weights from Weight SRAM and activations from Activation SRAM into the compute core, accumulating partial sums, and writing results back out to DRAM. Done naively, every partial product re-fetches operands from memory.

Memory access reduction

The computation and memory-access pattern can be changed to minimize cost without impacting the result. If W₀₁ is already loaded, arrange the schedule so every output that needs W₀₁ consumes it before it is evicted — one load, many uses.

Maximize reuse of loaded data

It is preferable to minimize memory access by maximizing the reuse of already-loaded operands. A weight held stationary in the compute core (recall the 200× DRAM penalty) amortizes its expensive fetch across many MACs. Scheduling neural-network layer accesses to minimize memory usage is developed further in the next lecture.

06

CNN accelerators: cost, im2col, tiling

The convolution is the core building block of a CNN and its most compute-intensive layer. Understanding its cost and how to reshape it into a matmul is the key to accelerating it.

Computational cost of a convolution

A conv maps input feature maps (B, C, H, W) through M filters of kernel size K×K to outputs (B, M, E, F). Iterating over seven dimensions — B, M, C, E, F, and the two kernel axes K, K:

MACs = B×M×K×K×C×E×Fmultiply-accumulates
Storage = 32 × (M·C·K·K + B·C·H·W + B·M·E·F) bitsweights + in-act + out-act, FP32
  • B batch · C input channels · H,W input map size · M filters · K kernel size · E,F output map size.

The 7-deep loop nest is the dataflow

for b, m, c, w(→E), h(→F), k₁, k₂: out[b][m][e][f] += in[b][c][e+k₁−(K+1)/2][f+k₂−(K+1)/2] · filter[m][c][k₁][k₂]the whole convolution
Dataflow = how you order the loops

This simple loop nest can be transformed in numerous ways to capture different reuse patterns of activations and weights, and to map the computation onto hardware. A CNN's dataflow defines how the loops are ordered, partitioned, and parallelized. A single scalar multiplier could execute the whole thing via this for-loop — slowly.

Scalar → vector → matrix machines

Scalar multiplier

z = xy. Most flexible, least efficient. The CPU/GPU per-element view.

Vector multiplier

z = xᵀy. A dot product per step — CPU & GPU SIMD lanes.

Matrix multiplier

Z = XY. Whole matmul as one op — the TPU model. Most efficient, least flexible.

The same flexibility↔efficiency trade returns: the scalar machine is maximally flexible; the matrix machine is maximally efficient.

im2col: turning convolution into GEMM

A standard convolution can be converted to a 2D matrix multiplication using the Im2Col operation. Each sliding receptive field is flattened into a row of an input matrix; filters flatten into columns:

[BEF × CK²] × [CK² × M] = [BEF × M]input matrix × filters = output

The reshaped input feature maps (rows = each output position, width K = Cᵢ·Hf·Wf) multiply the filter matrix to produce the output feature maps. This lets the mature, highly-tuned GEMM path (cuBLAS, systolic arrays) run the convolution — at the cost of duplicating overlapping pixels in the unrolled matrix.

Tiling

Decompose the big matmul

To handle matrix multiplication at large sizes, it is decomposed into tiles: in [W₀₀ W₀₁; W₁₀ W₁₁] × [X₀₀ X₀₁; …], each Wᵢⱼ and Xᵢⱼ can itself be a sub-matrix. Tiles are sized to fit on-chip SRAM so operands are loaded once and reused across the tile — directly attacking the DRAM cost.

07

Systolic arrays & DNN hardware

The dominant matmul engine: a 2D grid of MAC cells through which data rhythmically flows, each cell reusing a stationary weight so operands are fetched from memory only once.

The building blocks of a DNN processor

  • Processing engine — the MAC datapath.
  • Accumulator — sums partial products.
  • Register file — local operands.
  • Special function unit — activations and non-linear ops.
  • Memory subsystem — Weight SRAM, Data SRAM, DRAM, behind a memory controller with read/write control bits.

Spatial architecture & double buffering

Spatial computing

A grid of PEs fed from memory forms a spatial architecture. It can achieve great reuse of the extracted content — a weight W₁ loaded once is shared across many PEs and cycles — reducing memory access cost. This is data reuse realized in space rather than time.

Double buffering

Give the systolic array two memory banks (Mem₁, Mem₂). While the array consumes from one bank, DRAM fills the other; then they swap. Overlapping data production and consumption hides transfer latency and keeps the array continuously fed — no stalls waiting on DRAM.

The systolic array weight-stationary

  • Kung & Leiserson, "Systolic Arrays for VLSI" (1978) and Kung, "Why Systolic Architectures?" (1982).
  • A 2D grid of multiplier-accumulators (MACs) performing matrix multiplication.
  • Used by the Google TPU for deep learning (2017), among others.

Each systolic cell takes streaming inputs x (from below) and y (from the left), holds a stationary weight w in the MAC, and outputs:

z = w·x + y v = xMAC to the right · pass x up

Worked cell: with w=6, x=4, y=2 → z = 6·4 + 2 = 26 flows right, and v = x = 4 flows up to the next row.

How a full multiply runs

Weights are preloaded into the array (weight-stationary). Data enters skewed (diagonally staggered) from the edges so that each partial product w·x meets the correct incoming partial sum y at the right cycle. Partial sums accumulate as they march across the array, and after the pipeline fills, each cycle emits another column of the result matrix. For [2 −1 1; −2 3 −4] × [1 0; 0 3; −2 2], the skewed stream produces [0 −1; 6 1] — the whole GEMM computed with each weight fetched exactly once.

Why this wins

Every weight is loaded once and reused across the entire data stream; every partial sum stays on-chip, hopping cell-to-cell instead of round-tripping to DRAM. Combined with the 200× DRAM penalty, that reuse is exactly why systolic arrays are the heart of TPU-class accelerators.

Exam recap

The 5-layer stack

Framework → IR/graph-opt → ML compiler/codegen → kernel library → hardware. The IR is the analyzable graph the compiler rewrites.

Graph vs eager

Graph mode = build-then-run (optimizable). Eager mode = run-immediately (easy, but harder to optimize).

Non-diff ops

Subgradient, approximation, custom backward, or error. STE approximates the gradient as 1 so signal flows.

Graph rewrite

x*2+x*3 ≡ x*(2+3). torch.fx traces & transforms; e.g. drop redundant consecutive ReLU.

Kernel libs

cuDNN = DL-specific ops (conv, pooling, RNN). cuBLAS = general GEMM. Triton = Python GPU kernels with auto-tiling.

Flexibility ↔ efficiency

CPU → GPU → FPGA → CGRA → ASIC. Right = most efficient (TOPS/W), left = most flexible.

Energy hierarchy

ALU 1× · RF 1× · PE 2× · Buffer 6× · DRAM 200×. Moving data, not computing, is the cost.

Conv cost

MACs = B·M·K·K·C·E·F, seven nested loops. The loop ordering is the dataflow.

im2col + tiling

im2col reshapes conv → GEMM [BEF×CK²]·[CK²×M]. Tiling fits sub-matrices on-chip for reuse.

Systolic array

Kung 1978, TPU 2017. Cell: z=w·x+y, v=x. Stationary weights + skewed data = each operand fetched once.
